🔍 Catalogo Controlli di Qualità — 3 categorie
Puntuali
🎯
Agiscono sul singolo campo di un'entità. Verificano formato, tipo, vincoli e obbligatorietà.
GPD payment_date ✓ PASS
not_null · is_not_empty
valore: 2026-05-14
FDR amount ✗ FAIL
is_positive · max_value(9999999)
valore: -150.00 ← negativo
GEC status_code ⚠ WARN
is_in_set(['OK','KO','PENDING'])
valore: null → soglia warn 5%
Intra-entità
🔗
Valutano la coerenza interna tra attributi della stessa entità o tabella.
GPD date range ✓ PASS
end_date > start_date
2026-06-01 > 2026-01-15 ✓
FDR totale ✗ FAIL
total = sum(line_items)
atteso: 1500.00 — trovato: 1480.00
GEC stato ✓ PASS
if status=CLOSED → closed_at not null
closed_at: 2026-04-30 ✓
Cross-entità
🔀
Confrontano dati tra tabelle o domini diversi. I più complessi, i più preziosi.
GPD × FDR riconciliazione ✓ PASS
gpd.debt_id must exist in fdr.debt_id
referential integrity: 100% match
GPD × GEC importo ✗ FAIL
gpd.amount = gec.notified_amount
disallineamento su 23 record
Silver × Bronze completezza ⚠ WARN
row_count(silver) = row_count(bronze)
silver: 99.7% — soglia: 99.9%
✓ PASS check superato ⚠ WARN soglia warning superata ✗ FAIL check fallito — alert inviato

Standard BDD per la scrittura dei feature file — garantisce leggibilità, manutenibilità e stabilità dei test automatici.

REGOLA D'ORO
Scrivere scenari comprensibili anche a chi non conosce la funzionalità
REGOLA CARDINALE
Un comportamento = uno scenario — evitare coppie When-Then multiple
LESS IS MORE
Scenari brevi e concisi — ideale < 10 step per scenario
DICHIARATIVO ✓ vs IMPERATIVO ✗
✗ When l'utente clicca sulla casella
  And digita il numero avviso
  And preme Invio

✓ When l'utente inserisce il numero avviso

✗ Given l'utente apre Chrome e naviga su uat.checkout...

✓ Given la home page Checkout è visualizzata
GIVEN / WHEN / THEN — ANATOMIA
Given Precondizioni — stato iniziale (Setup)
When Azione centrale che scatena il comportamento
Then Verifica risultato atteso (Asserzione)
→ Sempre terza persona singolare · Presente indicativo · Ordine rigoroso
REGOLE DI STILE & FORMATTAZIONE
  • 1 blocco Feature per file · max 12 scenari
  • Lunghezza step: 80–120 caratteri
  • Keyword sempre capitalizzate (Given, When, Then, And, But)
  • Nessuna punteggiatura (punti/virgole) a fine step
  • Pipe | nelle tabelle Examples perfettamente allineate
  • Usare Scenario Outline + Examples invece di duplicare scenari
NOMENCLATURA & TAG STANDARD
  • Feature: descrive il modulo/dominio, non l'azione di test
  • Scenario: auto-esplicativo, < 15 parole, chi legge il log capisce subito
  • Tipologia: @smoke-test · @regression · @sanity
  • Livello: @e2e · @api · @integration
  • Dominio: @checkout · @backoffice · @auth-service
  • Tracciabilità: @PQ-XXX-CUP-2027
