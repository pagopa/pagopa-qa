Deep Dive · Test Automation Service

Due template pronti, complessità ridotta

Per i due strumenti più diffusi in Dipartimento esistono template ufficiali, pronti all'uso: pochissima configurazione, tutta la complessità resta nascosta dentro il TAS.

🔷

Azure DevOps

Un unico riferimento al repository TAS nella pipeline. Il risultato dei test compare direttamente nel tab "Tests" nativo di ADO.

🐙

GitHub Actions

Un singolo step nel workflow. Le metriche dei test (passati, falliti, tempo) sono subito disponibili senza parsing manuale.

Esistono anche opzioni più avanzate per casi particolari (es. testare un branch non ancora sul ramo principale), ma per la maggior parte dei team i template ufficiali bastano.
Deep Dive · Test Automation Service

Un responso chiaro per decidere se rilasciare

Al termine di ogni esecuzione, il TAS restituisce un resoconto strutturato: le pipeline lo usano per bloccare automaticamente un rilascio in caso di problemi.

Novità

L'AI lavora come il team QA

Stiamo dotando il nostro workflow di un sistema di agenti AI governati: non un singolo assistente generico, ma ruoli distinti che collaborano con regole chiare, proprio come un team QA reale.

Come funziona

Analyst
→
Engineer
→
Runner
→
Closer

Un orchestratore coordina 6 agenti specializzati: chi analizza i requisiti, chi scrive i test, chi li esegue e diagnostica i problemi, chi chiude con documentazione e pull request. Ognuno resta nel proprio perimetro. ↗ Vedi tutti i 6 ruoli

⚡ Più velocità

Attività ripetitive automatizzate, onboarding più rapido sulle convenzioni del team.

🎯 Meno errori

Ogni agente ha un ruolo specifico: meno modifiche fuori scope, meno variabilità.

📎 Tracciabilità

Regole e comportamenti sono versionati insieme al codice, non conoscenza tribale.

Novità

Un nuovo portale unico che raccoglie in un solo posto gli strumenti prima sparsi tra Dashboard One, Portale NRT e fogli di lavoro — evoluzione naturale di quanto già in uso.

🔍 Catalogo Controlli di Qualità — 3 categorie
Puntuali
🎯
Agiscono sul singolo campo di un'entità. Verificano formato, tipo, vincoli e obbligatorietà.
GPD payment_date ✓ PASS
not_null · is_not_empty
valore: 2026-05-14
FDR amount ✗ FAIL
is_positive · max_value(9999999)
valore: -150.00 ← negativo
GEC status_code ⚠ WARN
is_in_set(['OK','KO','PENDING'])
valore: null → soglia warn 5%
Intra-entità
🔗
Valutano la coerenza interna tra attributi della stessa entità o tabella.
GPD date range ✓ PASS
end_date > start_date
2026-06-01 > 2026-01-15 ✓
FDR totale ✗ FAIL
total = sum(line_items)
atteso: 1500.00 — trovato: 1480.00
GEC stato ✓ PASS
if status=CLOSED → closed_at not null
closed_at: 2026-04-30 ✓
Cross-entità
🔀
Confrontano dati tra tabelle o domini diversi. I più complessi, i più preziosi.
GPD × FDR riconciliazione ✓ PASS
gpd.debt_id must exist in fdr.debt_id
referential integrity: 100% match
GPD × GEC importo ✗ FAIL
gpd.amount = gec.notified_amount
disallineamento su 23 record
Silver × Bronze completezza ⚠ WARN
row_count(silver) = row_count(bronze)
silver: 99.7% — soglia: 99.9%
✓ PASS check superato ⚠ WARN soglia warning superata ✗ FAIL check fallito — alert inviato
